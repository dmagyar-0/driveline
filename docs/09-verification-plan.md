# 09 — Verification Plan

Driveline has three tiers of automated tests plus a short manual-
check list. This document describes what each tier covers and the
performance targets that gate any production release. The MVP
milestone gates have all been passed and have been removed from this
file; consult git history if you need the original M0–M6 ladder.

## Reference environment

- **Hardware:** a recent consumer laptop with H.264 hardware decode.
- **Browser:** latest stable Chrome or Edge.
- **OS:** macOS, Linux, or Windows.

Performance numbers below are measured on this class of machine. Lower
hardware is acceptable as long as functional checks pass and perf
degrades gracefully.

## Sample corpus

Generated by `sample-data/generate.py` (see Chapter 11 of the book).
All fixtures sit under `sample-data/` and are gitignored; expected
SHA-256 hashes are committed for drift detection.

- **`short.mcap`** — 10 s of 4K H.264 video on `/camera/front` plus
  three signal channels (`/vehicle/speed` scalar, `/imu/accel` vector,
  `/control/mode` enum).
- **`short.mf4`** — the same three signal channels, no video. Same
  `start_time` so overlays match the MCAP signals sample-for-sample.
- **`short.mp4` + `short.mp4.ts.bin`** — the same 10-second video as
  a standalone mp4 plus a 300-entry `i64` sidecar of absolute `log_time`s.
- **Reference PNGs** under `sample-data/refs/t_<ms>.png` for five
  known-frame times. Used by Playwright pixel-compare assertions.

## Automated tests

### Rust unit / integration (`cargo test --workspace`)

Covers every `Reader` implementation plus the fixture-generation
helpers. Representative tests:

- `reader::tests::infers_channel_kind_from_schema`
- `mcap::tests::builds_keyframe_index_from_fixture`
- `mcap::tests::fetch_range_respects_time_bounds`
- `mcap::tests::fetch_range_includes_prev_when_requested`
- `mf4::tests::translates_cg_time_to_ns_utc`
- `mf4::tests::fetch_range_returns_expected_arrow_schema`
- `mp4_sidecar::tests::rejects_length_mismatch`

Runs natively — no wasm toolchain needed. A full workspace run takes a
few seconds.

### JavaScript unit (`pnpm test:web` / vitest)

Under `environment: "node"`. Representative files:

- `state/store.test.ts` — clamp invariants, auto-pause, merge-source.
- `timeline/playback.test.ts` — anchor math and scrub-vs-tick
  distinction with deterministic `now`/`raf` stubs.
- `tests/arrow.contract.test.ts` — the Rust↔JS Arrow round-trip (see
  below).
- `panels/seriesFromArrow.test.ts`, `panels/mergeSeries.test.ts` —
  Arrow → typed-array conversions and k-way series merging.

### Contract test

One committed fixture (`test-fixtures/arrow_scalar.ipc`) that Rust
produces and JS parses. The check asserts exact row count, dtype, and
first/last timestamps. If either side upgrades Arrow independently,
this test fails before anything else can silently diverge.

### End-to-end (`pnpm e2e` / Playwright)

Runs the built app against headless Chromium with the sample corpus
served by the Vite dev middleware. Representative specs:

1. **File load** — drop `short.mcap`; channel list shows 4 channels;
   timeline spans 10 s.
2. **Scrub-and-assert** — set `cursorNs` to each of the five reference
   times and pixel-compare the video canvas to the reference PNG.
3. **Signal alignment** — min/max of rendered series match
   pre-computed expectations within the visible window.
4. **MF4 overlay** — drop `short.mf4` alongside the MCAP; same-named
   scalar series from both sources overlap within one sample.
5. **Playback** — click play; after 1.0 s of wall-clock, `cursorNs`
   advanced by `1.0e9 ± 50e6` ns.

## Manual checks

Human-in-the-loop, before any release:

- Drop files in unusual orders (mp4 before sidecar, MF4 before MCAP).
- Rapid scrub flicks — video settles on the final position, no stale
  half-seek.
- Play near the end of the session — loops or stops cleanly, no
  runaway.
- Tab backgrounded for 30 s then foregrounded — decoder recovers, no
  runaway CPU.
- Resizing a panel while playing — no visible glitches.
- Add a second PlotPanel via the `+` button — renders independently.

## Performance targets

Measured on the reference environment with the primary fixture. Use
`performance.mark` / `performance.measure` plus the `VideoDecoder`
queue stats for decoder back-pressure, and DevTools' Performance tab
for frame timing.

| Metric | Target |
|---|---|
| `open_file` (MCAP, 10 s 4K) | < 500 ms to channel list |
| `open_file` (MF4, 100 MB) | < 1 s to channel list |
| `fetch_range` for 10 s @ 1 kHz scalar | < 50 ms to Arrow bytes |
| Playback — dropped frames | < 1% over 10 s at 1× speed |
| Scrub seek settle | < 250 ms from drop to correct frame |
| Cursor tick end-to-end latency | < 16 ms (one rAF) |
| Signal panel render, 1 M points | deferred — decimation removed; revisit with LTTB/downsample if large logs become jittery |
| Memory after session load | < 300 MB RSS for the tab |

A regression in any of these is a blocking release issue. A
measurable improvement is worth preserving with a test where
feasible.

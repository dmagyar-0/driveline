# Daily Code Review — 2026-04-29

**Range:** `8e0b808..c99beae` (35 commits)

## Summary

A very large delta — Phases 2-10 of the V1 shell integration plus three
follow-up video-pacing fixes — landed in two days. The biggest concrete
regression is that bad-file drop errors now go to `/dev/null`: Phase 2
deleted the only UI that surfaced `openFiles().errors` (the
`SessionSummary` shim) and nothing replaced it, so a malformed MCAP/MF4
silently disappears for the user. Beyond that the work is solid: the 4K
pacing fix is well-reasoned, all new panels ship with unit tests, the
persist adapters fail closed, and panel ids are now centralised.
Smaller issues clustered around HTML correctness inside `EventsDrawer`
(input nested inside button), per-tick worker postMessage churn from
`VideoPanel`, and a few accessibility nits on the timeline markers.

## Findings

### Critical
None.

### High

- **`apps/web/src/App.tsx:285,468` — `openFiles().errors` is silently
  discarded.** Phase 2 removed `recentErrors` state and the
  `<ul data-testid="session-errors">` block alongside the
  `SessionSummary` shim. Both `useEffect`-side `onDrop` and the dev
  hook `openFiles` now `return await ...openFiles(files)` and toss
  `result.errors` on the floor. Nothing in the new `SourcesDrawer`
  (or anywhere else under `apps/web/src/shell/`) renders these
  errors, so a user who drags an unsupported file or a malformed
  `.mcap` gets no feedback at all.
  *Fix:* either surface drop errors in `SourcesDrawer` (a small
  errors section at the bottom seems natural) or push them through a
  toast/banner host. `BucketError` already carries `{ name, reason }`
  so the renderer is trivial; the missing piece is wiring the latest
  `result.errors` into the store (or local state) so a drawer can
  read it.

### Medium

- **`apps/web/src/panels/VideoPanel.tsx:351-360` — worker
  `setCursor(bigint)` fires on every cursor change.** The new
  `useSession.subscribe` callback runs unconditionally for every
  store mutation; whenever `cursorNs` differs from the previous tick
  (i.e. ~60 Hz during playback) it dispatches a Comlink
  `client.setCursor(...)` call. That is 60 cross-thread postMessages
  per second of playback, all to update a watermark whose only use is
  the `LOOKAHEAD_NS = 300 ms` decoder gate. The pacing logic only
  needs the latest value at the next refill — debouncing or only
  pushing when `(state.cursorNs - lastSentNs) > LOOKAHEAD_NS / 2`
  would cut the chatter by an order of magnitude with zero
  behavioural change.
  *Fix:* throttle to ≤ 10 Hz (or coalesce against `LOOKAHEAD_NS`) in
  the `subscribe` body before calling `client.setCursor`.

- **`apps/web/src/shell/drawers/EventsDrawer.tsx:155-207` — `<input>`
  nested inside `<button>`.** The bookmark row is a `<button>` whose
  child swaps between a `<span>` (label) and an `<input>` (rename).
  An interactive element inside a `<button>` is invalid HTML
  (`<button>` only allows phrasing content with no interactive
  descendants). The author tried to mitigate via
  `onClick={(e) => e.stopPropagation()}` on the input, but the
  structure still breaks Enter/Space activation: pressing Enter in
  the rename input commits via `onEditKey`, but a missed
  `stopPropagation` (e.g. `onBlur`) can re-fire the parent button's
  seek handler.
  *Fix:* lift the rename `<input>` out of the seek button — render
  the row as a flex container with a separate seek `<button>` and
  the rename `<input>`/`<span>` as siblings. Same pattern the
  remove-button already uses.

### Low

- **`apps/web/src/panels/{Map,Enum,Scene,Table}Panel.tsx` — same
  "drop binding when channel missing" pattern as PlotPanel.** All
  four new panels copy the existing `useEffect(() => { if
  (bindingId && channel === null) setBinding(panelId, null); })`
  guard. With persisted layouts, this fires *before* the user has
  loaded a fixture (channels list is empty on hydrate), wiping every
  persisted binding. The bug pre-exists in PlotPanel; Phase 6
  multiplies its blast radius across four more panel kinds. A proper
  fix is to gate the clear on "we have at least one source loaded"
  (`sources.length > 0` or `globalRange !== null`).

- **`apps/web/src/timeline/BookmarkMarkers.tsx:56` — `tabIndex={-1}`
  but `role="button"`.** The CSS file even ships a
  `:focus-visible` rule, so the design intends focus styling — but
  the JS opts out of the tab order. Either drop the role and the
  focus-visible CSS (markers are visual only) or set `tabIndex={0}`
  + Enter/Space handler so they're keyboard-reachable.

- **`apps/web/src/panels/TablePanel.tsx:230` — sub-millisecond
  precision is dropped before the float divide.**
  `Number(tsNs / 1_000_000n) / 1000` truncates the bigint at ms
  before converting; that's intentional per the comment, but the
  column header advertises `ts (s)` and the format is `.toFixed(3)`,
  i.e. exactly milliseconds — the comment says "6-digit precision"
  which the code doesn't actually provide. Either widen to
  microseconds (`tsNs / 1_000n`) and divide by 1e6, or update the
  comment.

- **`apps/web/src/panels/MapPanel.tsx:181` — third-party tile fetch
  per panel.** Each MapPanel hits
  `tile.openstreetmap.org` directly. Fine for a dev tool, but worth
  flagging that (a) every running browser leaks the user's IP to a
  public endpoint and (b) OSM's tile usage policy explicitly
  discourages production use without a fallback CDN. Not a
  blocker — note in `STATUS.md` and consider Stadia/MapTiler later.

### Nits

- **`apps/web/src/panels/PlotPanel.tsx:113-115` and
  `apps/web/src/panels/{Enum,cursor}Overlay.ts:29-34` — repeated
  `getComputedStyle(documentElement)` per cursor tick.** Three
  separate code paths now resolve `--color-accent-orange` /
  `--color-fg-2` / `--color-border-subtle` on every render. Cache
  once per panel mount (or memoise on `useEffect([])`) — saves a
  trivial amount of CPU on a hot path that is otherwise carefully
  budgeted.

- **`apps/web/src/state/store.ts:247-251` and
  `apps/web/src/state/store.ts:531-534` — duplicated `mintId`
  helper.** `mintBookmarkId` was lifted to a function but the
  identical `randomUUID`-with-fallback pattern in
  `saveCurrentLayoutAs` was inlined. Pull it into one helper.

- **`apps/web/src/state/store.ts:387-393` — `setLayoutJson` clears
  `activeNamedLayoutId` unconditionally.** The comment says the
  intent is "any out-of-band edit clears identity," which is right
  for `onModelChange`, but the same setter is also called by
  `restoreNamedLayout` *before* the active id is set in the same
  `set({...})` — fine because both fields land in one call, but
  worth a comment in `setLayoutJson` itself pointing at the seam,
  not just one in `restoreNamedLayout`.

- **`apps/web/src/layout/persist.ts:34-37` — schema bumps drop the
  user's old layout on upgrade.** v1 → v2 → v3 each return `null`
  from `validate()`, so anyone with a v2 payload (only Phase 5
  testers) loses their layout silently. Pre-v1 app stage means
  this is acceptable, but a one-time best-effort migration
  (`videoHudOn = {}`, empty new maps) would have been cheap.

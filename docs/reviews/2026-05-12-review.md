### Nits

- **`apps/web/src/panels/videoReadiness.test.ts:143-154` — existing
  "subscriber throws" test now leaks `console.warn` output to test
  stderr.** Commit `b5b0918` changed `scheduleNotify`'s catch to
  `console.warn("videoReadiness: subscriber threw", err)`. The new
  test at lines 156-178 mocks `console.warn`, but the older sibling
  test at 143-154 still installs a `vi.fn().mockImplementation(() =>
  { throw new Error("boom"); })` subscriber without a spy. Each
  vitest run will print a real "videoReadiness: subscriber threw
  Error: boom" line to stderr. Not a correctness bug, but it muddies
  CI logs and trains reviewers to ignore that warning.
  *Fix:* wrap the older test in the same `vi.spyOn(console,
  "warn").mockImplementation(() => {})` / `warn.mockRestore()` pair
  used by the new test, or hoist a `beforeEach` spy.

- **`apps/web/src/panels/videoReadiness.ts:47-52` — `console.warn` is
  unthrottled inside an rAF-coalesced notify.** Each rAF flush walks
  every listener and warns per throw. If a buggy Transport subscriber
  starts throwing on every readiness flip (the exact regression this
  log is meant to surface), playback will produce ~one warn per
  state change — fine in practice, but if a future caller subscribes
  something that throws on *every* notify (e.g. a dev hook), the log
  would fire at the rAF rate. Cosmetic; not worth fixing now, but
  worth a one-liner noting "if you ever see this in steady-state,
  add a once-per-listener gate" near the catch.

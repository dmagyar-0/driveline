### Findings I considered and discarded
- **`apps/web/src/panels/PlotPanel.tsx:351` — cursor-overlay effect listing `publishSync` as a dep.** The review agent flagged this as a re-render trigger, but `publishSync` is already `useCallback`'d at line 147 with a stable dep set, so the effect only reruns on real cursor/range/seriesKey changes. Not a bug.
- **`apps/web/src/panels/seriesFromArrow.ts:48` — `rawTsNs = rawTs.subarray(0, n)` aliasing the Arrow backing buffer.** Per ECMAScript §25.1, a `TypedArray` retains a strong reference to its underlying `ArrayBuffer`, so the Arrow storage can't be reclaimed while `rawTsNs` is live. The file's own comment (lines 44–47) calls this out and treats it as a deliberate zero-copy optimization. Not a bug.
- **`crates/data-core/src/mcap.rs:300–306` — keyframe vector grown without a cap.** The surrounding `McapReader::load` path already reads every message into `timestamps`, `values`, and `video_samples` Vecs in memory (lines 308–316); a per-channel keyframe cap wouldn't meaningfully bound the reader's memory use. The whole-file load is a documented MVP choice. Not actionable.
- **`crates/data-core/src/mp4_sidecar.rs:285–295` — keyframe-search fallback assumes monotonic `pts_ns`.** `pts_ns` is populated in file order at line 122 and the sidecar format specifies monotonic timestamps, so the assumption holds. Not a bug.
- **`apps/web/src/panels/VideoPanel.tsx:205–207` — potential double-close of `VideoFrame`.** Traced through the splice/shift sequence: `splice(0, blitIdx)` removes frames 0..blitIdx-1 (already closed in the prior loop), the target at original `blitIdx` ends up at `q[0]`, gets drawn, closed once, then shifted. No double-close.
- **`apps/web/src/timeline/playback.ts:79` — `Math.round(elapsedMs * 1e6 * a.speed)` loses sub-ns precision.** Rounding error is bounded at ½ ns per tick and the anchor is reset on speed change, so drift is immaterial for a tool whose source data is sampled in the µs–ms range.

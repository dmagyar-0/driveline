# Daily Code Review — 2026-05-02

**Range:** `09ed7bc..2d1a833` (24 commits, of which 9 carry source changes)

## Summary

A focused video-pipeline window. The headline is a full lazy-load refactor of
the mp4+sidecar path: WASM no longer holds sample bytes in linear memory, the
JS-side `Mp4SampleCache` reads `[offset, offset+size)` from the source `File`
on demand, and the `dataCore` worker is bypassed entirely for mp4 sources in
favour of a separate `mp4Lazy` MessagePort. On top of that landed five video
correctness fixes (paused-scrub frozen canvas, scrub-during-play frozen
canvas, ffmpeg-AUD AVC-mode breakage, x264 `repeat-headers=1` decode stall,
playback stall after 16-frame priming) plus a sidecar tolerance fix and a
canvas letterbox CSS one-liner. No critical or high findings; the main worry
is medium-severity documentation drift in `docs/05-video-pipeline.md`, which
still describes the eager-load architecture this window replaced. A few
defensive low/nit items below around malformed-mp4 underflows and per-element
wasm→JS calls in the new `mp4_sidecar_index` binding.

### Medium

- **`docs/05-video-pipeline.md` is materially out-of-date with the lazy-load
  refactor (`464ca00`, `e2f7c27`, `e2f63a0`).** This is the canonical
  architecture doc for the subsystem and now misleads on three fronts:
  - The Pipeline overview diagram (lines 19-26) still shows
    `EncodedChunk: data-core ───▶ video-decode`. For mp4 sources that arrow
    no longer exists — `videoDecode` reads sample bytes from a main-thread
    `Mp4SampleCache` over a separate `mp4Lazy` MessagePort
    (`apps/web/src/workers/videoDecodeOps.ts:41-52`,
    `apps/web/src/panels/VideoPanel.tsx:201-228`).
  - "Per-channel lifecycle" step 3 describes a single `video_open(...)`
    call returning AVCC `description` for both source kinds. Reality is now
    split by source kind: mcap runs Annex-B with a `findSps` codec probe
    (`videoDecode.worker.ts:159-194`); mp4 runs AVC mode with a synthetic
    `avcC` derived from the sidecar's SPS+PPS (`mp4AnnexB.ts:22-55`,
    `videoDecodeOps.ts:138-160`).
  - "Open flow" step 2 says `Mp4SidecarReader::open` "parses the mp4's
    moov." It now parses **only** `[ftyp][moov]`, sliced on the JS side by
    `readMp4HeaderBytes` (`apps/web/src/state/mp4HeaderSlice.ts:28`,
    `mp4_sidecar.rs:62-80`). The motivation (avoiding multi-GB
    `arrayBuffer()` on the main thread) is the entire point of the
    refactor and belongs in the doc.
  - The buffered-ranges and pending-fetch UI surfaces
    (`Transport.tsx:177-194,272-303`) are undocumented.
  Fix: replace the "mp4 + sidecar timestamp path" section with the new
  flow, and add a short paragraph contrasting AVC-mode (mp4) vs Annex-B
  (mcap). The trim-whitespace doc edit in `09588bb` shows the file is in
  active maintenance, so this drift is not a "haven't touched it" case.

### Low

- **`crates/data-core/src/mp4_sidecar.rs:272,279,289` — silent u32 underflow
  on malformed `stsc`.** Three sites compute `first_chunk - 1` /
  `chunk_id as usize - 1` against parsed mp4 fields without first checking
  `>= 1`. Per ISO/IEC 14496-12 §8.7.3 chunk and sample IDs are 1-based, so
  on well-formed mp4s this is fine; on a hand-crafted file with
  `stsc.entries[i+1].first_chunk == 0` the subtraction underflows (panic
  in debug, wrap to `u32::MAX` in release, then `chunk_offsets.get(usize
  ::MAX - 1)` returns `None` → `Mp4(InvalidData)`). The release-mode
  recovery is benign but not principled; the rest of the function
  surfaces `Mp4(InvalidData("..."))` for every other malformed-input
  case. Fix: add explicit `if first_chunk == 0 { return Err(...); }` and
  `entries[i+1].first_chunk.checked_sub(1)` guards. Driveline parses
  user-uploaded files, so defensive parsing matters here.

- **`apps/web/src/state/store.ts:441-454` — `setCursor` bumps `seekEpoch`
  unconditionally.** Even when the requested `ns` clamps to the current
  `cursorNs`, `seekEpoch` increments and the VideoPanel's
  `useSession.subscribe` listener (`VideoPanel.tsx:429-446`) schedules a
  50 ms debounced video seek. The seek's `lastSeekTargetRef` dedupe then
  no-ops the actual `client.seek(target)`, so the worst case is a wasted
  setTimeout cycle plus a frame-queue clear that didn't need to happen.
  This is reachable from any UI code that does a "reset cursor" /
  "snap to bookmark" / "go to current cursor" round-trip. Fix: short-
  circuit at the top, e.g. `if (clamped === cursorNs) return;` before
  bumping the epoch. (Cosmetic — would also skip a redundant React
  render, but this slice is read by enough panels that the saving is
  not nothing.)

- **`apps/web/src/workers/mp4AnnexB.ts:91,113` — `stripInlineParameterSets`
  breaks on a zero-length NAL instead of skipping.** The Rust ancestor
  `avcc_to_annexb` used `continue` for `nal_len == 0` (kept in
  `2026-05-01-review.md`'s baseline as a tested edge case). The JS port
  uses `break`, so a sample shaped `[len=0][SPS][PPS][AUD][slice]` would
  exit the first-pass scan with `needsStrip = false` and feed the SPS/PPS
  through to the decoder verbatim — the very thing this commit `e8fe195`
  was meant to prevent. In practice no encoder emits a leading zero-length
  AVCC unit, but the asymmetry with the Rust precedent (and with the
  in-test comment "truncation returns input verbatim", which is only true
  before any SPS/PPS) is worth straightening. Fix: change the two `break`s
  to `i += 4; continue;` to match the Rust loop.

### Nits

- **`crates/wasm-bindings/src/lib.rs:436-447` — per-element `set_index()`
  loop in `mp4_sidecar_index`.** The comment claims the manual loop
  avoids "an intermediate `Vec` per array of length N (videos can have
  ~1e5 samples)." But each `set_index(i, v)` is a wasm→JS cross-call.
  For 1e5 samples × 4 typed arrays that's 4×10^5 boundary crossings on
  open. `Uint32Array::copy_from(&[u32])` and `Uint8Array::copy_from(&[u8])`
  are direct memcpys and will be substantially faster. `BigInt64Array` /
  `BigUint64Array` don't have that path (BigInts can't memcpy), but we
  can build a `Vec<u64>` then `BigInt64Array::view_mut_raw` /
  `unsafe { js_sys::BigInt64Array::view(...) }` — see the `wasm-bindgen`
  cookbook for the safe-vs-fast tradeoff. Not urgent for the 10 s 4K
  fixtures (300 samples) but real recordings will be 10^4–10^5 samples.

- **`apps/web/src/timeline/Transport.tsx:175-192` — `bufferedSegments` is
  rebuilt inside the component body every render.** The Transport
  re-renders on every `cursorNs` tick (60 Hz during playback). The
  computation is cheap for the current "few resident runs per source"
  shape, but it's pure of cursor state and would naturally live inside a
  `useMemo([loadedRanges, globalRange])`. Today the render churns N
  divisions and a few `bigint` subtractions per tick.

- **`apps/web/src/panels/VideoPanel.tsx:195-200` — `findMp4Cache(handle)`
  does an `Array.find` per `mp4Sample` call, i.e. per decoded sample.**
  At 30 fps this is 30 linear scans of `sources` per panel per second.
  With `sources.length` typically 1–3 it's invisible, but the bridge is
  trivially closed-over so the cache reference could be hoisted once
  per stream. Ignore unless the source list ever grows past tens.

- **`apps/web/src/state/mp4SampleCache.ts:38` — `lastUsedTick` is a
  module-level counter shared across every `Mp4SampleCache` instance.**
  Each cache only ever compares its own entries' ticks against itself, so
  there's no correctness issue — but the Number value monotonically
  increases for the lifetime of the page and would lose monotonicity
  somewhere past `Number.MAX_SAFE_INTEGER`. Realistically unreachable;
  flagged only because instance-local counters are a one-line change and
  remove a class of "bug or not?" question.

- **`apps/web/src/panels/VideoPanel.module.css:12-14` — `object-fit:
  contain` on a `<canvas>` quietly relies on the canvas's intrinsic
  dimensions being set on first decode (`VideoPanel.tsx:146-148`).** That
  invariant holds — `sizedRef` flips after the first frame — but if
  someone removes the size assignment `object-fit` becomes a no-op (the
  default canvas intrinsic size is 300×150). A one-line CSS comment
  pointing at `sizedRef` would help.

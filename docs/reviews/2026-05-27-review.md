### Medium

- **`scripts/convert_comma2k19_to_mf4.py:230-231` vs
  `scripts/convert_comma2k19_to_mcap.py`** — inconsistent behaviour when
  `--only` resolves to no samples. The MF4 converter exits with
  `--only filtered out every group; nothing to write`, but only if
  `n == 0` after summing the enabled groups; if `enabled={"gnss"}` and
  every GNSS row is NaN it still exits. The MCAP converter under the
  symmetric case (e.g. `--only speed` on a segment whose
  `processed_log__CAN__speed__t` is empty) silently writes an MCAP with
  a registered channel and zero messages. Pick one policy and apply it
  to both — either both refuse empty output, or both emit it. The
  silent-empty path is easy to mis-diagnose downstream when a wrapper
  script `--only`'s a group that the upstream parquet variant doesn't
  carry. **Fix:** in `convert_comma2k19_to_mcap.py`, count appended
  `entries` per enabled group and `sys.exit` with a clear message if
  every enabled group produced zero entries (mirror the MF4
  guard).

### Nits

- **`scripts/convert_comma2k19_to_mf4.py:117-119`** — `t_lists`
  builds a numpy array for every `__t` column in the segment, even
  when `--only` will discard most groups. The cost is small relative
  to parquet load, and using the global anchor across runs is
  intentional, so the simpler code is fine — but a one-line comment
  explaining "anchor uses ALL groups for cross-run consistency"
  would prevent a future maintainer from "optimising" it to scan
  only `enabled` groups and breaking the MCAP↔MF4 clock alignment.
- **`.claude/skills/verify-comma2k19/SKILL.md:3`** — `allowed-tools`
  is space-separated (`Bash(...) Bash(...) ...`); other skills in the
  tree (none in this repo to compare against, but the convention in
  upstream examples) use comma-separated lists. May or may not parse
  the same way in the Claude Code skill loader — worth verifying once.
- **`apps/e2e/tests/_demo-comma2k19-video.spec.ts:42`** —
  `VIDEO_NATIVE_ID = "1/video"` is a hard-coded native channel id
  shared across tests with different source ordering (test 1 loads
  2 sources, test 5 loads 5). It works only because
  `findChannelId({sourceName, nativeId})` finds the source by
  `sourceName` first and then resolves the native id against that
  source's channel list (`apps/web/src/App.tsx:410-418`). Spelled out
  in a one-line comment, it stops being a foot-gun.
- **`.claude/skills/verify-comma2k19/scripts/fetch-sources.sh:29-30`**
  — `curl -fL ... -o "$out.partial"` plus `mv "$out.partial" "$out"`
  is fine, but a curl failure under `set -e` leaves a stale
  `.partial` next to the destination. The idempotent skip on the
  next run uses `[[ -s "$out" ]]` (the real path) so it retries
  correctly, but a `trap 'rm -f "$out.partial"' ERR` inside
  `download()` would keep `$DEST` tidy.
- **5 binary screenshots checked in (~1.3 MB total)**. The SKILL.md
  documents them as the "last-good outputs … so a regression shows
  up in the diff." They aren't pixel-compared in the spec — they
  exist purely for PR-time visual inspection. That is a reasonable
  trade, but flagging it so the next reviewer doesn't expect
  `toMatchSnapshot()` semantics.

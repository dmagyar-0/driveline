### Low

1. **`apps/web/src/panels/mergeSeries.ts:152-176` — closure indirection on
   the hot path.** `nextX` and `consume` are arrow closures created inside
   `mergeStepHold` that capture `inputs`, `cursorIdx`, `markerPhase`,
   `marker1`, `marker2`. Each output sample pays `2k` calls into `nextX`
   (once for `minX` discovery, once for the match) plus up to `k` calls into
   `consume`. V8 generally inlines small monomorphic closures, but when this
   is called from the plot hot path with k=8–12 series and N≈1M samples, the
   call overhead can dominate the actual arithmetic. **Fix:** inline both
   helpers into the main loop as plain reads/writes against the typed-array
   state — same shape, no function-call overhead, and no `Number.isNaN`
   guards (replace the NaN sentinel with an explicit `exhausted` boolean
   per input or just track an `activeCount`). Worth doing only if a perf
   regression shows up; the algorithm itself is fine.

### Nits

1. **`apps/web/src/panels/mergeSeries.ts:128-135` — upper-bound is up to
   3·N·k.** `upper += xs.length; +2 per gap` means the output buffer is
   sized for the worst case of every interval being a gap. This matches
   the old code's allocation behaviour (push-then-sort), so it isn't a
   regression — but the comment "Worst case (every interval > threshold)
   adds 2 × samples" undersells it: it's "adds 2 × (samples − 1) per
   input". A one-line tightening of the comment would save a future reader
   from a stare-and-recount.

2. **`scripts/convert_comma2k19_to_mcap.py:257-267` — warning prints once per
   triggering segment, which can be noisy in batch.** A multi-segment
   batch convert against the comma2k19 corpus may emit dozens of identical
   warnings to stderr. The warning *is* the right surface and the message
   is useful, but consider gating it behind a `--verbose` flag or
   collapsing to "warned N times" at script exit if this script is run
   in CI / batch.

3. **`crates/data-core/src/mcap.rs:1259-1273` — idempotence test is solid;
   one micro-improvement.** The `assert_ne!(once, zstd, …)` check is a
   nice fast-fail guard. The `assert_eq!(once, twice, …)` could be
   tightened by also asserting `once.len() > zstd.len()` (decompression
   should always grow), which would catch a regression where the first
   pass somehow returned the input verbatim *and* the second pass also
   returned it verbatim — a different failure mode than `assert_ne!`
   already covers.

**1. `crates/data-core/src/mcap.rs:429-434` — `debug_assert_eq!` tripwire is a tautology**

```rust
let stream = ::mcap::MessageStream::new(&owned)?;
debug_assert_eq!(
    std::any::type_name_of_val(&stream),
    std::any::type_name::<::mcap::MessageStream>(),
    "predecompress_zstd_chunks output is only safe with mcap::MessageStream — a future \
     swap to an indexed reader must rewrite/strip stale offsets in the pre-pass first",
);
```

The intent (per the doc comment, line 257-275) is correct and important: the
predecompress pre-pass leaves stale byte offsets in `MessageIndex` /
`ChunkIndex` / `SummaryOffset` records, so only a linear reader is safe. But
this assertion compares `type_name_of_val(&stream)` against
`type_name::<::mcap::MessageStream>()`. Both are resolved at compile time. A
future engineer who swaps the LHS (`let stream = …IndexedReader::new(&owned)?`)
will have to re-think the call site anyway — `for msg_result in stream` would
fail to compile if the iterator item type changed, and likely require local
edits regardless. So the tripwire only "fires" against the very narrow case of
a refactor that mechanically renames the constructor without touching the
nearby block, which is implausible.

**Concrete fix:** delete the `debug_assert_eq!` and lean on the rust-doc
`# Precondition` block already added at lines 257-275. If you want a real
compile-time tripwire, add a private trait `LinearReaderOnly` and bound the
return of an internal helper on it:

```rust
trait LinearReaderOnly {}
impl<'a> LinearReaderOnly for ::mcap::MessageStream<'a> {}
fn open_linear_stream(buf: &[u8]) -> Result<impl LinearReaderOnly + Iterator<…>> { … }
```

Severity is low because the current code is harmless (the assert always
passes) and the doc comment is the real safety net.

**2. `apps/web/src/panels/mergeSeries.ts:124-185` — `mergeStepHold` perf is materially worse than `mergeUnion` on large series**

The default-mode `mergeUnion` (lines 65-110) is a k-way merge with one cursor
per input — `O(N · k)` total, no allocation other than the output `Float64Array`s.

`mergeStepHold` (the new gap-threshold path) instead:
1. Pushes all xs from every input into a plain `number[]` (`candidate`),
2. Plus 2 entries per detected intra-series gap,
3. Calls `Array.prototype.sort((a,b)=>a-b)` on it (O(N log N), and
   non-monomorphic comparator),
4. Then dedupes into a `Float64Array`.

For a comma2k19-scale plot panel (a few hundred thousand samples per series,
two series, so ~500k entries) this is roughly an order of magnitude slower
than the k-way merge path. It runs on the main thread inside the
`PlotPanel.tsx:340-396` data-fetching effect on every range update.

This is opt-in (default is `null` → `mergeUnion`), so it's a LOW, not a HIGH —
but worth fixing before anyone ships a "use step-hold mode by default"
recommendation.

**Concrete fix:** replace the candidate `number[]` + sort with a k-way merge
that's identical to `mergeUnion`'s walker, but additionally tracks
intra-series gaps and emits the held-end + gap-start markers inline as it
advances. The cursor state for marker emission is per-input, just like the
existing `idx[]`. This keeps the path `O(N · k)` with one Float64Array
allocation.

**3. `apps/web/src/panels/mergeSeries.ts:144-148` — comment overstates floating-point exactness**

```js
// Float comparison is exact here because all values came from input.xs
// (Float64) or arithmetic on those values, never from user-typed thresholds.
```

This is wrong as a general claim. `lastX + threshold` and
`lastX + threshold + epsilon` are IEEE-754 ops and may produce values that
don't compare strictly equal to other arithmetic combinations — e.g., if two
different gap markers happen to be computed from `lastX_a + threshold` and
`lastX_b + threshold` with `lastX_a == lastX_b` but in different rounding
contexts (none here, but the *guarantee* the comment claims doesn't hold in
general). The actual property is "values equal under `===` will dedupe;
values unequal under `===` won't, but that's at worst an extra sub-epsilon
point in xs, not a correctness bug."

**Concrete fix:** replace the comment with: "Strict `!==` is fine: false
negatives just leave a sub-`epsilon`-spaced extra entry in xs, which is
visually indistinguishable from the held-end marker. Strict false positives
can't happen for distinct inputs since IEEE-754 equality is reflexive."

**N2. `scripts/convert_comma2k19_to_mcap.py:163-170` — lazy RR drops samples silently if first sample has 3 wheels**

If the first 4 wheel-speed samples have `len(vec) == 3` and the fifth has
`len(vec) == 4`, the channel is registered on sample 5 and samples 1-4 emit no
RR value at all. The `wheel_speed_rr` channel will start later than its
siblings. That's correct behavior given the design (no synthesis), but it's
surprising — a downstream consumer expecting four parallel time series gets
ragged starts. A one-line WARN log when transitioning from "no RR" to
"registered RR mid-stream" would help future debugging.

**Range:** `7ddcb44..f90cf42` (20 commits)

## Summary
First-pass review covering the entire repo from the initial design-doc commit through M2 T2.4 (session registry + file drop). The most material issue is worker lifecycle in `apps/web/src/App.tsx`: the `useEffect` that constructs the two Comlink workers has an empty deps array and **no cleanup function**, so under React 18 `<StrictMode>` (enabled in `main.tsx`) dev-mode double-invocation spawns four workers and leaks two. A handful of medium/low robustness issues round out the list (silent `i64 → i32` truncation in MCAP enum decoding, swallowed errors in `useSession.clear()`, dependence on Arrow internals from the dev hooks). Rust readers, Arrow IPC contracts, and test coverage are otherwise solid.

## Findings

### Critical
_None._

### High
- **`apps/web/src/App.tsx:103-163` — Worker leak under StrictMode / on remount.** The effect creates `dataCore`/`videoDecode` workers and installs `window.__drivelineDevHooks`, but the effect has empty deps and **returns no cleanup**. With `<StrictMode>` (`apps/web/src/main.tsx:9`), React 18 mounts → unmounts → remounts every effect in dev, so two pairs of workers are spawned and one pair is permanently orphaned. On every refresh you also reassign `window.__drivelineDevHooks` to a closure capturing the *new* refs, while the previous workers stay alive holding their own thread + WASM heap. **Fix:** return a cleanup that calls `dataCore.current?.[Comlink.releaseProxy]?.()`, terminates the underlying `Worker`, clears `useSession.getState().setWorker(null)`, and deletes `window.__drivelineDevHooks`. Also have `makeDataCoreClient` / `makeVideoDecodeClient` return `{ proxy, worker }` so callers can `terminate()` the raw `Worker`.

### Medium
- **`apps/web/src/workerClient.ts:26-40` — No termination affordance.** Both factories construct a raw `Worker` and immediately wrap it; the `Worker` reference is dropped, so callers cannot `terminate()` it even if they want to. Same fix as above: return both the proxy and the underlying `Worker`.
- **`apps/web/src/state/store.ts:247-266` — `clear()` swallows close errors silently.** The `catch { /* swallow */ }` block hides legitimate worker faults (panics in the WASM closer, dropped postMessage replies, etc.) while still resetting state — so the UI shows an empty session even though slab handles may still be live in the worker. **Fix:** at minimum `console.warn("close failed", s.kind, s.handle, err)`; ideally surface the error in `recentErrors` or a similar diagnostic channel.
- **`crates/data-core/src/mcap.rs:162-165` — `as i32` silent truncation on enum codes.** `parse_enum_json` does `f.as_i64().map(|i| i as i32)`; an out-of-range JSON `value` (e.g. `0x1_0000_0000`) silently truncates to `0`, fabricating a valid-looking enum sample from malformed input. Untrusted MCAP payloads make this a real (if unlikely) corruption path. **Fix:** `i32::try_from(i).ok()` and drop the sample on overflow.

### Low
- **`apps/web/src/App.tsx:108-161` — Dev-hook closures use non-null assertions on refs.** Every method does `dataCore.current!.…` / `videoDecode.current!.…`. Today nothing ever clears those refs, but pair this with the missing cleanup above and a future refactor that nulls a ref will silently call `undefined.foo`. **Fix:** add `if (!dataCore.current) throw new Error("dev hook used before init")` once at the top of each closure, or capture the proxy in a const inside the effect.
- **`apps/web/src/App.tsx:127-147` (and `:111-118`) — Reads Arrow internals (`table.getChild("ts")!`, `value.get(i)`).** These are dev-hook-only paths so the blast radius is small, but `table.getChild("ts")!` will throw a confusing `cannot read properties of null` if the IPC schema drifts. **Fix:** explicit error: `const ts = table.getChild("ts"); if (!ts) throw new Error("missing ts column");`.
- **`crates/data-core/src/mcap.rs:246` — `topic.clone()` per message in the hot loop.** `channel_id` is cloned twice per message (line 246 and again line 248 for the `entry()` lookup) when an `Arc<str>` or holding `topic` as `&str` and only cloning into the map on insert would be enough. Fine for short fixtures, wasteful at production scale. **Fix:** make `ChannelId = Arc<str>`, or look up the entry by `&str` and clone only on insert.
- **`crates/data-core/src/mcap.rs:299-321` — Per-channel resort allocates 3 vectors and rebuilds the `values` Vec.** Acceptable for MVP; worth a `// TODO(T4.3): decimation will resort, drop this once that lands` comment so it's clearly temporary.
- **`apps/web/src/state/store.test.ts:139` — Undocumented `@ts-expect-error`.** Suppresses the legitimate type error that `__setBlocking` is not on `Remote<DataCoreApi>`. Add a one-line comment explaining it's a fake-worker-only escape hatch.

### Nits
- **`apps/web/src/App.module.css:62-67` — `.errors { color: #b23a00; }` is a fixed light-mode color.** With `prefers-color-scheme: dark`, error text becomes nearly invisible on the dark background. Use `light-dark()` or a CSS variable.
- **`crates/data-core/src/mp4_sidecar.rs:83` — `expect("8-byte chunk")` after `chunks_exact(8)`.** Provably unreachable, but mixing `expect` here with `unwrap()` elsewhere on the same idiom (e.g. the FourCC parsing in `fixtures.rs`) is inconsistent. Pick one style.
- **`crates/wasm-bindings/src/lib.rs:55, 148, 245` — `as u32` on slab handles.** Unreachable in practice (you'd need >4B concurrent readers in a wasm process), but worth a one-line comment noting the bound rather than silently truncating.

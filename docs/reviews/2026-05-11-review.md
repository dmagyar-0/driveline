  *Fix:* rewrite the comment to describe the autodetect path; drop the
  "expected to fail" claim.

### Nits

- **`apps/web/src/timeline/Transport.tsx:24` — single-element readonly
  array used with `.includes`.** `STATES_THAT_SHOW_DOT` only ever
  contains `"waiting"`, so `anyPanelWaiting` could just check
  `r.state === "waiting"`. The array shape signals more flexibility
  than the code actually delivers; if you want to keep the array, at
  least add a comment that "stalled" deliberately stays out. *Fix:*
  inline the equality check, or document the design choice.

- **`apps/web/src/workers/videoDecodeOps.ts:178` — `mp4MarkPending`
  fires unconditionally on `open()`, before the LRU check.** The
  comment acknowledges this is intentional ("fire BEFORE the sample
  fetch below so the spinner is up while the slice is in flight"),
  but when the first sample is already cached the spinner flips on
  and off within a microtask. Visually it is a no-op; cosmetic at
  worst. *Fix:* skip `mp4MarkPending` when `mp4Sample` would resolve
  synchronously from the LRU, or accept the noise.

- **`apps/web/src/timeline/playback.ts:38` — module-level
  `cursorGated` is process-global.** Only one playback loop runs at
  a time, so this is fine in production. Unit tests in
  `playback.test.ts` already exploit the global to assert gating;
  worth a sentence noting that `startPlaybackLoop` cannot be safely
  instantiated twice against the same module instance.

- **`apps/web/src/panels/videoReadiness.ts:43-50` —
  rAF-coalesced notify catches subscriber exceptions but discards
  them silently.** The `try { l() } catch {}` is correct (a buggy
  subscriber should not break the registry), but a noisy
  `console.warn` would help when a subscriber regression shows up in
  field telemetry. *Fix:* log the swallowed error.

- **`apps/e2e/tests/decodeAwareCursor.spec.ts:233` — scenario A's
  `advancedSec >= 2` over an 11 s window only proves cursor moved at
  18 % of real-time.** The test comment is explicit that the real
  load-bearing check is `frameIndex > 100`, but the assertion as
  written would also pass for a stream where the gate kept the
  cursor pinned for 8 s and then released. Consider tightening once
  CI hardware is faster, or asserting that the gate was *off* for
  the majority of the window via repeated `getCursorGated()` polls.

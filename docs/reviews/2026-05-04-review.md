## Summary
Solid range. The headline change is a `predecompress_zstd_chunks` pre-pass in
`crates/data-core/src/mcap.rs` that lets the wasm reader open
zstd-compressed MCAP files without dragging in `zstd-sys`; it works, is
well-tested, but ignores stale `MessageIndex` byte offsets inside rewritten
chunks — currently harmless because the reader is linear, but a future move
to indexed reads would silently return wrong messages and the new comment
should call that out as a precondition rather than an aside. The
comma2k19 converter quietly fakes a fourth wheel-speed channel by
duplicating `rl→rr`, which is fine in a private fixture but misleading in a
publicly-documented "real-world" demo. A `spanGaps: true` flip in
`PlotPanel.tsx` fixes the multi-mailbox CAN regression but also hides
genuine channel-loss gaps as horizontal hold lines. PR #79's
`qualifiedChannelId` work and the matching `VideoPanelContainer` fix are
clean and well-tested.

### Medium

- **`crates/data-core/src/mcap.rs:255-380` — `predecompress_zstd_chunks`
  leaves `MessageIndex` byte offsets inside rewritten chunks pointing at
  the old (compressed) record positions.** The doc-comment dismisses
  stale `ChunkIndex` / `SummaryOffset` offsets as "harmless because
  McapReader never dereferences them", and that's true *today* because
  `mcap.rs:405` consumes the file via `MessageStream::new(&owned)`
  (linear scan, ignores all indexes). But the same logic applies to
  `MessageIndex` records that live *inside* each chunk and point to
  message offsets within the chunk's records body — those offsets are
  now wrong by the compression delta and there is no comment about it.
  If anyone later wires up `IndexedReader` (or upstream `mcap` switches
  the default reader), zstd files will silently return garbage. **Fix:**
  either strip `MessageIndex` records (op 0x07) from rewritten chunks
  during the pre-pass, or convert the `// stale offsets are harmless`
  comment into a hard precondition (`// SAFETY: only safe while open()
  uses MessageStream — see mcap.rs:405`) and add a `debug_assert!` next
  to the `MessageStream::new` call so a future swap fails loudly.

- **`scripts/convert_comma2k19_to_mcap.py:174-180` synthesises a
  rear-right wheel-speed sample by duplicating `rl` when only three
  wheels are present.** This is documented in
  `sample-data/realworld/README.md` but the duplication is invisible at
  the MCAP level — anyone who downloads `comma2k19.mcap` and computes
  `wheel_speed_rl - wheel_speed_rr` gets exactly zero and a wrong
  conclusion about the vehicle. The e2e spec also doesn't assert
  anything about `wheel_speed_rr`, so the fakery isn't load-bearing.
  **Fix:** when `len(vec) < 4`, skip emitting `ch_wheel_rr` entirely;
  the panel "needs four traces" justification is cosmetic and the demo
  still works with three wheels.

- **`scripts/convert_comma2k19_to_mcap.py:117-119` — `t_min`'s NaN
  guard only checks `v[0]`.** `min(v)` with a NaN later in the list is
  undefined in CPython (`min([1.0, float('nan'), 2.0])` returns `1.0`
  *only* because of evaluation order; reordering the inputs flips the
  answer). With comma2k19 GNSS lists routinely starting clean and
  containing NaNs in the middle, `t_min` can come out as NaN and every
  emitted timestamp becomes garbage. **Fix:** filter inside the inner
  `min`, e.g. `min(x for x in v if not np.isnan(x))`, or pre-filter
  `t_lists` with `np.asarray(v)[~np.isnan(v)]`.

### Low

- **`apps/web/src/panels/PlotPanel.tsx:298 — `spanGaps: true` hides
  real channel-loss gaps as horizontal step-holds.** The fix is the
  right call for the multi-mailbox CAN bug it solves, and the comment
  honestly notes the trade-off, but in practice an operator looking at
  a plot can no longer distinguish "this channel stopped broadcasting
  for 5 s" from "this channel is just slower than its neighbour". The
  comma2k19 path doesn't trigger that case, but real-world MF4 logs do
  (e.g. CAN bus arbitration loss). Consider, in a follow-up, surfacing
  a per-channel "no data" overlay or a configurable gap threshold so
  the visual can stay step-held but real dropouts are still
  discoverable.

- **`apps/e2e/tests/realworld-comma2k19.spec.ts:62-79` pins
  sample-count expectations against a Hugging Face URL with no hash
  pin.** The spec depends on `demo-00000-of-00003.parquet` from
  `huggingface.co/datasets/commaai/comma2k19`. If the dataset
  maintainer reshards or updates that file, `speed.sampleCount >
  4_000` quietly stops being true and the spec becomes flaky months
  from now with no obvious cause. **Fix:** record the parquet's
  expected SHA256 in `sample-data/EXPECTED_HASHES.txt` (next to the
  others) and have `convert_comma2k19_to_mcap.py` verify it on read.

- **`crates/data-core/src/mcap.rs:282 — `Vec::with_capacity(input.len())`
  under-allocates when chunks decompress.** Compressed → uncompressed
  rewrite always grows; the output buffer reallocates at least once.
  Trivial perf nit (one realloc per `open`), but the comment promises
  "cheap on native too" and this is the easy half. Use
  `input.len() * 2` or accumulate a `decompressed_total` estimate.

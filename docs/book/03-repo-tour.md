# Chapter 3 — A Tour of the Repository

Open the repository root in your editor and you'll see roughly this:

```
driveline/
├── apps/
│   ├── web/            # The React + TypeScript browser app
│   └── e2e/            # Playwright end-to-end tests
├── crates/
│   ├── data-core/      # Portable Rust reader core
│   └── wasm-bindings/  # Rust ↔ JavaScript bridge
├── docs/               # The doc set (including this book)
├── sample-data/        # Scripts and fixtures for test data
├── test-fixtures/      # Small committed binaries shared Rust↔JS
├── Cargo.toml          # Rust workspace manifest
├── package.json        # JavaScript workspace manifest
├── pnpm-workspace.yaml # pnpm workspace declaration
├── rust-toolchain.toml # Pins the Rust compiler version
└── README.md
```

This chapter walks each top-level directory and the most important
files inside. You don't need to remember all of this — it's a map you
can flip back to while reading later chapters.

## `apps/web` — the browser app

Everything the user actually sees lives under
[`apps/web/src`](../../apps/web/src).

```
apps/web/
├── index.html                  # The HTML shell Vite serves
├── vite.config.ts              # Build / dev-server config (see ch. 11)
├── tsconfig.json               # TypeScript compiler options
├── package.json                # App's own dependencies
└── src/
    ├── main.tsx                # Entry point: mounts <App/>
    ├── App.tsx                 # Root component
    ├── App.module.css          # Its styles (CSS modules — scoped)
    ├── unsupportedSplash.ts    # "Your browser lacks WebCodecs" screen
    ├── workerClient.ts         # Spawns the two web workers
    ├── perf.ts                 # performance.mark/measure wrappers
    ├── wasm/                   # Generated — output of `pnpm wasm:build`
    ├── state/
    │   ├── store.ts            # Zustand store (session + transport + layout)
    │   └── bucket.ts           # Drop-target file type routing
    ├── layout/
    │   ├── Workspace.tsx       # FlexLayout container
    │   ├── panelFactory.tsx    # Maps layout IDs to React components
    │   ├── persist.ts          # localStorage layout persistence
    │   └── defaultLayout.ts    # 50/50 video+plot starting layout
    ├── panels/
    │   ├── VideoPanel.tsx      # Video playback (WebCodecs)
    │   ├── VideoPanelContainer.tsx # Wrapper that resolves bindings
    │   ├── PlotPanel.tsx       # Signal plot (uPlot)
    │   ├── ChannelPicker.tsx   # Tree-view channel selector
    │   ├── seriesFromArrow.ts  # Arrow IPC → typed arrays
    │   ├── mergeSeries.ts      # K-way merge of signal series
    │   ├── palette.ts          # Deterministic series colours
    │   └── cursorOverlay.ts    # Cursor line over the plot
    ├── timeline/
    │   ├── Transport.tsx       # Scrubber + play/pause/speed UI
    │   ├── playback.ts         # rAF-driven cursor advance
    │   └── formatTime.ts       # Human-readable timestamps
    ├── workers/
    │   ├── dataCore.worker.ts   # Main WASM worker
    │   └── videoDecode.worker.ts # WebCodecs worker
    └── tests/
        └── arrow.contract.test.ts
```

A few things worth noting before you open any of those files:

- **No HTML templates** other than `index.html`. React components
  describe their own markup in JSX.
- Files ending in `.module.css` are **CSS modules** — styles that are
  automatically scoped to a specific component so class names can't
  collide.
- The `wasm/` folder is generated by `pnpm wasm:build`. Never edit it
  by hand.
- Tests live next to the code they test (`foo.ts` ↔ `foo.test.ts`).
  The convention is that *unit* tests stay in `src/`; *end-to-end*
  tests live one level up under `apps/e2e/`.

## `apps/e2e` — end-to-end tests

Playwright is a browser-automation tool. `apps/e2e` contains specs
that launch a headless Chromium, open Driveline with real fixture
files, and assert that the page behaves correctly. Each `.spec.ts` file
is one scenario — dropping files, scrubbing, pixel-comparing rendered
video, asserting that cursor math stays within a tolerance.

You do not need to read these to understand the app, but they're the
closest thing to "worked examples of the whole stack being driven." If
you want to see what the dev-only hooks in `App.tsx` are *for*,
Playwright specs are the answer.

## `crates/data-core` — the portable Rust core

```
crates/data-core/
├── Cargo.toml       # Declares Rust dependencies (see ch. 11)
├── src/
│   ├── lib.rs       # Crate root — re-exports public API
│   ├── types.rs     # Channel, TimeRange, ChannelKind, DType, ...
│   ├── reader.rs    # The `Reader` trait itself
│   ├── noop.rs      # A trivial Reader used in tests
│   ├── mcap.rs      # McapReader — reads .mcap files
│   ├── mf4.rs       # Mf4Reader — reads .mf4 files
│   ├── mp4_sidecar.rs # Mp4SidecarReader — reads .mp4 + .mp4.timestamps
│   └── fixtures.rs  # Test-data generators
├── tests/           # Integration tests
└── examples/        # Small programs that exercise the library
```

This crate is "portable" in the sense that it contains no
browser-specific code. You can run `cargo test -p data-core` on your
laptop and it builds and runs natively. Chapter 4 reads the important
parts.

## `crates/wasm-bindings` — the bridge

```
crates/wasm-bindings/
├── Cargo.toml
└── src/
    └── lib.rs     # wasm-bindgen surface: open_mf4, mf4_summary, ...
```

One file. It depends on `data-core` and on the `wasm-bindgen` crate,
and it exposes a handful of functions the JavaScript side can call.
Chapter 5 covers how this works.

The reason it's a separate crate is so that `data-core` stays
compilable on a normal desktop (for fast tests) while only
`wasm-bindings` is ever built for `wasm32-unknown-unknown` — a
specialised compile target that produces a `.wasm` file.

## `sample-data/` — the fixture generator

A Python script
([`sample-data/generate.py`](../../sample-data/generate.py))
that produces the sample corpus used by tests: a 10-second 4K video
plus matching signal streams, written out as an MCAP, an MF4, and an
MP4+sidecar triple. The corpus itself is gitignored (large) but
expected SHA-256 hashes are committed so you can verify you regenerated
it correctly. Chapter 11 explains how to run it.

## `test-fixtures/` — committed small binaries

Small binary fixtures that both the Rust tests and the JavaScript
tests need in order to agree on a format. The key one is
[`arrow_scalar.ipc`](../../test-fixtures/arrow_scalar.ipc), the "round-
trip" fixture that proves Rust's Arrow IPC output parses identically
in the JS `apache-arrow` library.

## Files at the root

- **`Cargo.toml`** — the workspace manifest. Declares both Rust
  crates as members, pins a single workspace version, and sets release
  profile flags for small WASM output (`opt-level = "s"`, `lto = true`).
- **`Cargo.lock`** — records the exact resolved dependency versions
  for reproducible builds. **Commit it.**
- **`package.json`** — top-level scripts (`pnpm dev`, `pnpm build`,
  `pnpm wasm:build`, `pnpm test`, `pnpm e2e`). Does not declare any
  dependencies itself — each workspace member (`apps/web`, `apps/e2e`)
  owns its own.
- **`pnpm-workspace.yaml`** — tells pnpm which sub-packages form the
  workspace.
- **`pnpm-lock.yaml`** — the JS-side lockfile. Also commit.
- **`rust-toolchain.toml`** — pins the Rust toolchain channel.
  Running any `cargo` command in this repo will auto-install a matching
  `rustc` if you use `rustup`.
- **`Makefile`** — a thin convenience over the other scripts. Most
  targets shell into `pnpm` or `cargo`.
- **`.gitattributes` / `.gitignore`** — standard file-level config.
  `sample-data/` outputs are ignored except for small committed
  fixtures.
- **`README.md`** — the quickstart. Prerequisites, three commands to
  run, and a pointer to `docs/`.

## The size, at a glance

The codebase as of this book's writing is roughly:

- Rust: ~5 000 lines across `data-core` and `wasm-bindings`.
- TypeScript / TSX: ~6 000 lines in `apps/web/src`.
- Tests: ~1 line of tests per 6 lines of production code.

Most files are small. The biggest individual ones are `crates/data-
core/src/mcap.rs` (~1 200 lines — MCAP is a complex format) and
`apps/web/src/panels/PlotPanel.tsx` (~500 lines — lots of plot/UI
state to juggle). Everything else is under 500 lines.

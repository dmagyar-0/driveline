# Chapter 8 — The Wire Formats: Arrow IPC and Encoded Chunks

Two kinds of data cross from Rust to JavaScript in Driveline: columns
of numbers (signals) and opaque compressed frames (video). They don't
travel the same way. This chapter explains what each one looks like on
the wire, and why Driveline chose these representations.

## Signals: Apache Arrow IPC

### Why Arrow?

A plot panel needs to draw maybe 250 000 points at 60 Hz. The naive
path — serialise each `{ ts, value }` as JSON, parse on the JS side,
convert every string to a number — is measured in seconds, not
milliseconds.

**Apache Arrow** is a columnar in-memory format with a stable,
language-independent binary encoding. Its key property is that you can
hand a JavaScript `Float64Array` the same raw bytes Rust wrote out,
and read them directly. No parsing, no per-element work — a memcpy
away from "data in UI".

Driveline uses the **Arrow IPC file format**, which prepends a short
header to a batch of columns and lets the reader index straight into
the buffers.

### The Rust side

Every `*_fetch_range` function in `data-core` builds an Arrow
`RecordBatch` with a schema like this (from `mf4.rs:193`):

```rust
let schema = Arc::new(Schema::new(vec![
    Field::new(
        "ts",
        DataType::Timestamp(TimeUnit::Nanosecond, Some("UTC".into())),
        false,
    ),
    Field::new("value", DataType::Float64, false),
]));
```

Two columns, both non-nullable:

- `ts` — 64-bit nanosecond timestamps, typed as Arrow's
  `Timestamp(Nanosecond, "UTC")`. The timezone tag is part of the
  Arrow schema, not a separate convention — it tells every consumer
  that these are UTC.
- `value` — 64-bit floats. Every scalar channel, whatever its native
  dtype, widens to `f64` before shipping. Chapter 4 mentioned this:
  one numeric type everywhere keeps the Arrow schema uniform and saves
  the panels from branching on dtype.

The batch is written to a `Vec<u8>` with Arrow's `FileWriter`:

```rust
let mut buf = Vec::new();
let mut w = FileWriter::try_new(&mut buf, &schema)?;
w.write(&batch)?;
w.finish()?;
// buf now holds the IPC-encoded bytes.
```

Those bytes are returned as `ArrowIpc = Vec<u8>`, travel through the
wasm boundary as a `Uint8Array` (Chapter 5), cross the worker
boundary through Comlink (Chapter 7), and land in a React panel.

### The JavaScript side

`apache-arrow` (the official JS library) reads the same bytes:

```ts
import { tableFromIPC } from "apache-arrow";

export function seriesFromArrow(bytes: Uint8Array): PlotSeries {
  const table = tableFromIPC(bytes);
  if (table.numRows === 0) return EMPTY;

  const tsCol = table.getChild("ts")!;
  const valueCol = table.getChild("value")!;

  // Access the raw backing buffers directly — don't iterate.
  const rawTs = tsCol.data[0].values as BigInt64Array;
  const rawY  = valueCol.data[0].values as Float64Array;
  const n = Math.min(rawTs.length, rawY.length);

  const xs = new Float64Array(n);
  for (let i = 0; i < n; i++) xs[i] = Number(rawTs[i]) / 1e9;   // ns → seconds
  const ys = new Float64Array(rawY.subarray(0, n));
  const rawTsNs = rawTs.subarray(0, n);                         // zero-copy view
  return { xs, ys, rawTsNs };
}
```

Things to notice:

- `tsCol.data[0].values` is the actual `BigInt64Array` backing the
  timestamp column. There is no decode step — the bytes came from the
  worker's memory in exactly this layout.
- `.get(i)` would work but is slower and would convert each timestamp
  to a JavaScript `Date` object, quietly dropping sub-millisecond
  precision. The comment at the top of this file pins that down; the
  direct-buffer path is deliberate.
- ns → seconds conversion happens because uPlot (Chapter 10) wants
  numeric seconds for its x-axis. uPlot's x-axis is a plain number, so
  crossing `Number.MAX_SAFE_INTEGER` would start to lose precision —
  but seconds since the epoch is safely in range.
- `rawTsNs` is retained as a full-precision `BigInt64Array` view
  because the cursor-alignment snapshot (Chapter 10) needs to binary-
  search in exact nanoseconds.

### The round-trip contract

Arrow promises that Rust-written bytes parse identically in any
language's Arrow library. Driveline proves it with a **committed
fixture**: `test-fixtures/arrow_scalar.ipc` is a known Arrow batch
generated by the Rust code and checked into the repo. A JS test
(`apps/web/src/tests/arrow.contract.test.ts`) loads those bytes and
asserts the values come out exactly as expected.

If anyone ever upgrades the Arrow version on one side without updating
the other, this test fails loudly. It's one of the cheapest, highest-
signal safety nets in the repo.

## Video: Encoded Chunks

### Why not Arrow here?

Video doesn't fit the columnar model at all. A frame is a blob of
bytes whose size varies by content and whose meaning depends on which
frames preceded it. There are no columns to store, no numeric types
to widen, and decoding happens *in the browser*, not in Rust. What
the decoder wants is the same thing the container stored: compressed
H.264 access units, one at a time.

### The `EncodedChunk` shape

Rust defines it (simplified):

```rust
pub struct EncodedChunk {
    pub pts_ns: i64,
    pub is_keyframe: bool,
    pub data: Vec<u8>,
}
```

A timestamp, a keyframe flag, and the raw compressed bytes. That's the
whole wire format.

`EncodedChunkIter` is an iterator of these, returned by
`Reader::video_stream(...)`. The wasm bridge stores the iterator in a
second slab (Chapter 5), then pulls batches on demand.

On the JS side, the same shape appears in `dataCore.worker.ts`:

```ts
export interface EncodedChunkWire {
  pts_ns: bigint;
  is_keyframe: boolean;
  data: Uint8Array;
}
```

### Annex-B framing

H.264 bitstreams have two standard envelopes:

- **Annex-B**, where NAL units (the elementary H.264 pieces) are
  separated by a `00 00 00 01` start code. This is what lives inside
  `.ts`/MPEG-TS streams and what MCAP's `foxglove.CompressedVideo`
  payload stores.
- **AVCC**, where NAL units are prefixed with a 4-byte length. This is
  what `.mp4` stores *inside* the container.

WebCodecs' `VideoDecoder` accepts either, but the mode must match
what the decoder was configured with. The two sources use different
paths:

- **MCAP** frames arrive as Annex-B. The videoDecode worker scans the
  first keyframe for an SPS NAL unit, derives the `avc1.XXXXXX` codec
  string from it, and configures the decoder in Annex-B mode (no
  `description` field).
- **MP4** frames stay in AVCC (length-prefixed) form. The wasm layer
  returns SPS and PPS bytes via `mp4_sidecar_index`; `mp4AnnexB.ts`
  synthesises an `AVCDecoderConfigurationRecord` (`avcC`) from them.
  The decoder is configured in AVC mode with that record as the
  `description`, so raw mp4 sample bytes feed straight to `decode()`
  without any Annex-B conversion.

### Timestamp precision, carefully

WebCodecs uses microseconds (`EncodedVideoChunk.timestamp: number`),
but Driveline works natively in nanoseconds. The videoDecode worker
converts only when it has to, and keeps the original `bigint` for the
round-trip:

```ts
function ptsToMicros(ptsNs: bigint): number {
  // Lose sub-microsecond resolution but preserve frame ordering.
  return Number(ptsNs / 1000n);
}

// ... later, converting back for the cursor comparison:
const ptsNs = BigInt(frame.timestamp) * 1000n;
```

The worker reconstructs a `bigint` nanosecond PTS from the decoded
frame's microsecond timestamp. Every MCAP and MP4 fixture in
Driveline's sample corpus uses integer microsecond timestamps anyway,
so this conversion is lossless in practice.

### Batched pulls

The videoDecode worker does not pull one chunk per call — that would
turn the decoder-feeding loop into a chatty back-and-forth across the
MessagePort bridge (Chapter 7). Instead:

```ts
const PULL_BATCH = 8;
const REFILL_LOW_WATER = 4;

async function pullAndFeed(): Promise<void> {
  // ...
  const batch = await session.ops.next(session.streamId, PULL_BATCH);
  for (const c of batch) {
    session.decoder.decode(new EncodedVideoChunk({ ... }));
    session.inFlight += 1;
  }
}
```

Pull 8, feed 8, and when `inFlight` drops below 4 fetch another 8. The
decoder is kept full without any one call crossing the thread
boundary more than every few hundred milliseconds.

## Why two formats?

Could Arrow carry video? In principle — you could define a schema
with `pts: Timestamp(ns, UTC)` and `frame: Binary`. In practice it's
all cost: the extra Arrow schema overhead, the loss of direct typed-
array access (because frames have variable length), and no benefit
over the 3-field `EncodedChunk`. Simpler wins.

Could `EncodedChunk` carry signals? You'd give up columnar access.
Plot panels read `xs` and `ys` as two flat buffers; fetching
per-sample from a list of structs would make large batches measurably
slower.

So: Arrow for columns, `EncodedChunk` for blobs. The boundary is
exactly where the data model changes from "grid of numbers" to "stream
of opaque chunks."

## Recap

- Signals go as Arrow IPC record batches with a `ts`/`value` schema.
  The JS side reads the raw buffers without per-element work, and a
  committed fixture asserts the two sides stay compatible.
- Video goes as `EncodedChunk { pts_ns, is_keyframe, data }`. MCAP
  chunks are Annex-B framed; MP4 chunks are AVCC-framed (raw mp4
  samples) with the decoder configured via a synthesised `avcC`
  description. Both paths pull in batches to keep the decoder fed
  with minimal RPC traffic.
- Nanoseconds everywhere the data flows — widened to microseconds
  only when WebCodecs forces it.

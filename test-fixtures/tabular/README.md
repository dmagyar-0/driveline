# Tabular fixtures (CSV / Parquet)

Synthetic, deterministic fixtures for the **TabularReader** (CSV + Parquet
signal-log ingestion). Generated by `sample-data/generate_tabular.py`; re-run
`make fixtures` to regenerate and `make fixtures-check` to verify they are
byte-identical. Drag-and-drop demo copies live in `sample-data/tabular/`.

These fixtures deliberately exercise **both time bases** the reader's import
dialog must support:

- **Absolute epoch** ŌĆö the time column is microseconds since the Unix epoch.
- **Relative-to-clip-start** ŌĆö the time column starts near 0 and must be
  rebased onto an absolute epoch (the Alpamayo camera-frame convention).

All logs share the epoch anchor `START_US = 1_704_067_200_000_000`
(`2024-01-01T00:00:00Z`, `= 1_704_067_200_000_000_000` ns), matching
`sample-data/generate.py`'s `START_NS`, so they line up with the video corpus.

## Files

### `signals_abs.csv` / `signals_abs.parquet`

A "normal" vehicle signal log. Same schema and data in both encodings.

| column         | type           | unit                       | time basis |
| -------------- | -------------- | -------------------------- | ---------- |
| `t_us`         | int64          | microseconds since epoch   | ABSOLUTE   |
| `speed_mps`    | float64        | m/s                        | ŌĆö          |
| `steering_deg` | float64        | degrees                    | ŌĆö          |
| `rpm`          | float64        | revolutions per minute     | ŌĆö          |

30 rows @ 10 Hz (100 000 us spacing, ~3 s). Speed is a raised-cosine ramp
0 -> 15 m/s, steering a +/-12 deg sine (2 s period), rpm = 800 + speed * 180.

### `camera_frames_rel.csv`

Alpamayo-**camera**-style frame index table. The timestamp is **relative to
clip start** (first frame at 0).

| column         | type  | unit                              | time basis |
| -------------- | ----- | --------------------------------- | ---------- |
| `frame_index`  | int64 | frame counter (0-based)           | ŌĆö          |
| `timestamp_us` | int64 | microseconds since **clip start** | RELATIVE   |

30 frames @ ~30 fps (33 333 us spacing, matching the video corpus FRAME_NS).

### `egomotion_abs.parquet`

Alpamayo-**egomotion**-style log. The timestamp is **absolute epoch**
microseconds (note: differs from the camera table above ŌĆö this is the whole
point of the dual-time-basis test).

| column      | type    | unit                     | time basis |
| ----------- | ------- | ------------------------ | ---------- |
| `timestamp` | int64   | microseconds since epoch | ABSOLUTE   |
| `vx`        | float64 | m/s (longitudinal)       | ŌĆö          |
| `vy`        | float64 | m/s (lateral)            | ŌĆö          |
| `yaw_rate`  | float64 | rad/s                    | ŌĆö          |

30 rows @ 50 Hz (20 000 us spacing).

## Expected `TimeBasis` in the import dialog

What a tester should select for each file to get correct alignment. `unit` is
the source column's unit; the reader converts to nanoseconds internally.
`epoch_offset_ns` is the nanoseconds added to the (converted) source time.

| file                    | time_column    | unit         | mode     | epoch_offset_ns                                   |
| ----------------------- | -------------- | ------------ | -------- | ------------------------------------------------- |
| `signals_abs.csv`       | `t_us`         | microseconds | absolute | 0                                                 |
| `signals_abs.parquet`   | `t_us`         | microseconds | absolute | 0                                                 |
| `camera_frames_rel.csv` | `timestamp_us` | microseconds | relative | 1_704_067_200_000_000_000 (= START_US * 1000)     |
| `egomotion_abs.parquet` | `timestamp`    | microseconds | absolute | 0                                                 |

For the `relative` file, the reader should add the clip-start epoch
(`epoch_offset_ns`) to each rebased timestamp; absolute files need no offset.

## Reproducibility

Parquet is written with `compression="snappy"`, `version="2.6"`,
`write_statistics=False`, `store_schema=False`, `use_dictionary=False`,
`data_page_version="2.0"` so re-running the generator yields byte-identical
files. CSV uses explicit `LF` line endings. `make fixtures-check` regenerates
into a temp dir and byte-compares.

Caveat: the Parquet footer embeds a `created_by` string carrying the pyarrow
version (currently `parquet-cpp-arrow version 24.0.0`). Output is byte-stable
for a fixed pyarrow version; a pyarrow upgrade will change those bytes and
require regenerating the committed fixtures (same as the other generated
fixtures in this repo).
